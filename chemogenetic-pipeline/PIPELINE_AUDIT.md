### MODULE 4: Molecular Docking

**Status: BROKEN — Vina not installed, all docking scores are fake**

What's real:
- Receptor preparation is genuine: PDB 8E9W (real hM3Dq cryo-EM structure) was downloaded from RCSB, cleaned with PDBFixer, converted to PDBQT
- Ligand preparation is genuine: RDKit 3D conformers + Meeko PDBQT conversion
- All receptor/ligand PDBQT files exist and are valid

What's broken:

1. **AutoDock Vina is not installed.** `from vina import Vina` fails with ImportError. The code falls back to `_estimate_docking()` which computes: `affinity = -5.0 - (atom_count / 10.0) + np.random.normal(0, 0.5)`. This is just a formula based on atom count plus random noise — it has zero relationship to actual binding.

2. **Every row in `docking_results.csv` has `estimated=True`.** The affinities (-7.77 for CNO, -7.21 for DCZ, etc.) are meaningless numbers.

3. **No docked poses exist.** The `data/docking/poses/` directory doesn't exist because Vina never ran.

   **Fix:** Install Vina and re-run Module 4.
   ```bash
   pip install vina
   python3 -m src.module4.run_module4
   ```

   Note: Vina requires the receptor in PDBQT format (already done) and the binding site coordinates. The code hardcodes center=(-10.0, 5.0, -15.0) and box size=(25, 25, 25). These should be verified against the actual binding pocket of 8E9W.

4. **Binding site coordinates may be wrong.** The center (-10.0, 5.0, -15.0) is hardcoded. This should be validated by checking where the co-crystallized ligand sits in PDB 8E9W. If these coordinates are off, Vina will dock into the wrong part of the protein and produce meaningless results even when it's running for real.

   **Fix:** Verify binding site by examining the ligand coordinates in 8E9W. The orthosteric binding pocket of muscarinic receptors is well-characterized.

**Summary of fixes needed:**
- [ ] `pip install vina`
- [ ] Verify binding site coordinates against 8E9W crystal ligand position
- [ ] Re-run Module 4: `python3 -m src.module4.run_module4`
- [ ] Confirm `estimated=False` in output and that docked pose PDBQTs are generated

### MODULE 5: Virtual Screening

**Status: BROKEN — fake compound library, fake docking scores**

What's real:
- Tanimoto similarity calculations use real RDKit Morgan fingerprints — this part is correct
- The hit ranking/filtering logic is sound

What's broken:

1. **The compound library is fake.** `download_zinc_library()` does NOT query the ZINC database. Instead it generates 151 compounds from 26 hardcoded seed SMILES by doing string manipulations (e.g., `smiles.replace("N", "O")`). This produces chemically dubious molecules — some are disconnected fragments (e.g., "F.c1ccc2c(c1)..." which is a mixture of fluoride ion + a tricyclic).

   **Fix:** Replace with a real compound library. Options:
   - Download a ZINC subset for CNS-active compounds from https://zinc20.docking.org
   - Query PubChem for compounds similar to known DREADD actuators using the PubChem PUG-REST API
   - Use the ChEMBL client (already installed) to pull compounds with measured activity at muscarinic receptors

2. **All screening docking scores are fake.** Since Module 4's `dock_compound()` uses the heuristic fallback, all screening affinities are also `estimated=True`.

   **Fix:** After installing Vina (Module 4 fix), re-run Module 5.

3. **The top "hit" (ZINC_000125) is a disconnected fragment.** SMILES: `F.c1ccc2c(c1)N=C(N1CCNCC1)c1ccccc1N2` — the `F.` prefix means it's a mixture of fluoride ion and a separate molecule. This is an artifact of the string manipulation approach to library generation.

   **Fix:** Implement a real library download/query function. At minimum, add a SMILES validity check that rejects disconnected fragments.

**Summary of fixes needed:**
- [ ] Replace `download_zinc_library()` with real database query (ZINC, PubChem, or ChEMBL)
- [ ] Add SMILES validation to reject disconnected fragments
- [ ] Install Vina first (Module 4 dependency)

### MODULE 6: ADMET Prediction

**Status: PARTIALLY BROKEN — BBB model is real, other 4 models are trained on garbage**

What's real:
- **BBB model:** Trained on the actual BBBP dataset (Martins et al., 2050 compounds) downloaded from DeepChem. Uses RandomForest + GradientBoosting ensemble on 2048-bit Morgan fingerprints. The model IS functional — it correctly scores caffeine (known BBB+) at 0.907 and glucuronic acid (known BBB-) at 0.488. However, AUROC on scaffold-split test set is only 0.5, meaning it doesn't generalize well to structurally novel compounds.
- Morgan fingerprint generation is real RDKit

What's broken:

1. **hERG, CYP2D6, HIA models are trained on synthetic data.** Because PyTDC is not installed, the code calls `_generate_synthetic_dataset()` which creates 800 "compounds" by repeating 10 drug-like SMILES labeled as positive and 8 non-drug SMILES labeled as negative, 40-50 copies each. This is not training data — it's the same molecules repeated.

2. **hERG = CYP2D6 = HIA.** Because all three models are trained on the same synthetic scaffolds with the same logic, they produce IDENTICAL predictions for every compound. DCZ gets 0.8675 from all three. They are literally the same model.

3. **Clearance model is also trained on synthetic data.** R² = -0.003, MAE = 18.8 — worse than predicting the mean. Completely useless.

   **Fix:** Install PyTDC to get real training data:
   ```bash
   pip install PyTDC
   ```
   PyTDC provides real curated datasets:
   - BBB: B3DB (7,807 compounds) or BBBP (already cached)
   - hERG: hERG central (>300K measurements)
   - CYP2D6: TDC CYP2D6 substrate/inhibitor datasets (13,130 compounds)
   - HIA: Human Intestinal Absorption dataset
   - Clearance: Hepatocyte/microsomal clearance datasets

   Then re-run Module 6 to retrain all models on real data.

4. **BBB model generalization is poor (AUROC 0.5 on scaffold split).** Even the "good" model doesn't generalize. Consider:
   - Using a larger dataset (B3DB has 7,807 compounds vs BBBP's 2,050)
   - Adding more features beyond Morgan fingerprints (e.g., RDKit descriptors, physicochemical properties)
   - Using a different split strategy or cross-validation

**Summary of fixes needed:**
- [ ] `pip install PyTDC`
- [ ] Delete existing fake model files: `models/admet/herg_model.joblib`, `cyp2d6_model.joblib`, `hia_model.joblib`, `clearance_model.joblib`
- [ ] Re-run Module 6: `python3 -m src.module6.run_module6`
- [ ] Verify that hERG, CYP2D6, HIA now produce DIFFERENT predictions (not identical)
- [ ] Consider retraining BBB on the larger B3DB dataset

### MODULE 8: Molecular Dynamics

**Status: PARTIALLY WORKING — simulation ran for DCZ, but only DCZ**

What's real:
- OpenMM is installed (8.4) and genuinely executed
- System setup used real AMBER14 force field, TIP3P water, 0.15M NaCl
- `DCZ_solvated.pdb` (28 MB) — real solvated protein system
- `DCZ_system.xml` (88 MB) — real OpenMM serialized system
- `DCZ_prod.dcd` (62 MB) — real trajectory file
- `DCZ_prod.log` shows physically correct thermodynamics: ~300K temperature, ~-5,990,000 kJ/mol potential energy, ~3600 nm³ box volume for a ~360,904 atom system
- MDAnalysis is installed (2.10.0) for trajectory analysis

What's broken:

1. **`openmmforcefields` is not installed.** This package provides `GAFFTemplateGenerator` for ligand parameterization with the GAFF2 force field. If the existing DCZ simulation was set up when this was installed and it was later removed, new simulations for other compounds will fail at the system setup step.

   **Fix:**
   ```bash
   pip install openmmforcefields
   ```

2. **Only DCZ was simulated.** The other 5 compounds (CNO, Clozapine, Compound 21, Olanzapine, Perlapine) have no MD data.

   **Fix:** Run Module 8 for remaining compounds. Each simulation will take significant time depending on the number of steps configured.

3. **Module was interrupted mid-analysis.** Progress file shows step 3/5 "Analyzing trajectory" — the trajectory analysis may not have completed. If `_load_synthetic_analysis()` was called as fallback, the analysis CSV would contain fake random-walk RMSD data.

   **Fix:** Check if `data/results/md_analysis_DCZ.csv` exists and whether its data looks real (RMSD values should be in the 1-5 Å range for a stable GPCR, not a monotonically increasing random walk). Re-run analysis if synthetic.

4. **No membrane embedding.** The code solvates the protein in a water box but does NOT embed it in a lipid bilayer. For a GPCR (membrane protein), this is a significant limitation — the transmembrane domains would normally be embedded in a POPC/POPE lipid membrane. Running a GPCR in pure water can cause unrealistic unfolding of transmembrane helices.

   **Fix:** This is a major architectural change. Would need to use CHARMM-GUI Membrane Builder or OpenMMDL to properly embed the receptor in a lipid bilayer. For now, document this as a known limitation.

**Summary of fixes needed:**
- [ ] `pip install openmmforcefields`
- [ ] Verify DCZ analysis CSV is real data (not random walk)
- [ ] Run MD for remaining 5 compounds
- [ ] Document the no-membrane limitation (GPCR simulated in water only)
